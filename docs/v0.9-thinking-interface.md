# v0.9 Thinking Interface

v0.9 shifts Doings Evidence MCP from a research critique tool toward a practical thinking interface for organizational reasoning and client-safe language.

## Product position

The server should help Doings users:

1. think more clearly about organizational problems,
2. avoid solution-first consulting language,
3. translate evidence into language that can be used in client work,
4. write in a Doings voice: plain-spoken, evidence-honest, concrete and non-hype.

It is still not a systematic literature review tool. It is a critical evidence scanner and thinking partner.

## New default tool

### `think_with_evidence`

Use this as the default entry point for natural prompts such as:

- “Can we say this?”
- “How should I think about this?”
- “Make this more Doings.”
- “What is weak in this reasoning?”
- “Is there research behind this or does it just sound good?”
- “Make this client-safe.”

The tool combines:

- intent-aware critique from v0.8,
- argument mapping,
- solution-first detection,
- consulting-language risk detection,
- Doings voice rewriting,
- client-safe language,
- evidence-to-language translation,
- learning nudges.

Example:

```json
{
  "input": "We think the client needs a flatter organization to become faster.",
  "context": "Nordic professional-services company, 50-150 employees, project-based client delivery",
  "mode": "thinking_partner"
}
```

Expected behavior:

- do not jump to validating “flat is faster” as a standalone claim,
- first map the argument,
- flag solution-first framing,
- ask where speed is actually lost,
- produce a client-safe version and an executive version.

## New supporting tools

### `analyze_org_argument`

Maps a piece of reasoning into:

- symptom,
- diagnosis,
- assumed cause,
- proposed solution,
- mechanism,
- desired outcome,
- missing links,
- stronger diagnostic questions.

### `detect_solution_first_thinking`

Flags when a text names a solution before the underlying problem and mechanism are established.

Examples of solution-first signals:

- autonomous teams,
- squads/tribes,
- flatter organization,
- matrix,
- operating model,
- governance,
- OKRs,
- decision rights,
- removing middle management.

### `rewrite_in_doings_voice`

Rewrites a rough claim in a Doings voice:

- plain-spoken,
- evidence-honest,
- non-hype,
- concrete mechanisms over slogans,
- comfortable with uncertainty.

### `make_client_safe`

Turns a rough or internally critical statement into language usable in client dialogue.

It returns:

- internal critique,
- client-safe version,
- executive version,
- phrases to avoid.

## Design principle

v0.9 should not only answer. It should improve how the user thinks.

For risky consulting language, it should return a learning nudge such as:

```json
{
  "thinkingPattern": "vague causal consulting claim",
  "whyItMatters": "It names a desired outcome but hides the mechanism, level of analysis and boundary conditions.",
  "betterMentalModel": "State what changes, for whom, at what level, under which conditions, and compared with what alternative."
}
```

## Important limitation

This layer improves reasoning and writing, but it does not replace client discovery. A solution-first warning is a prompt to investigate, not a final diagnosis.
